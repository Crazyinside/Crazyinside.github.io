线程：

线程（英语：thread）是操作系统能够进行运算调度的最小单位。它被包含在进程之中，是进程中的实际运作单位。一条线程指的是进程中一个单一顺序的控制流，一个进程中可以并发多个线程，每条线程并行执行不同的任务。在Unix System V及SunOS中也被称为轻量进程（lightweight processes），但轻量进程更多指内核线程（kernel thread），而把用户线程（user thread）称为线程。

一个进程可以有很多线程，每条线程并行执行不同的任务。进程就像一个部门，一个线程则是部门内部的一个员工，他们并发的做着相同或不同的事来完成安排的任务。

malloc函数在内存中有一个内存空间 堆的概念，不是数据结构的堆。进程是一个程序执行的过程，堆就是一个进程的地址空间，每个进程都有一个默认的堆，malloc函数便是在进程的堆中再申请一个私有堆。申请的私有堆空间在使用完不会被自动释放。

如上代码，在堆中申请了10字节的私有堆，但是用来指向该私有堆地址的指针，是被存放到栈中的。

当一个函数被调用时，会将当前位置的指针压栈，然后再寻找函数入口执行函数，函数调用完时，直接出栈就跳回到了进入函数前的位置，然后带着函数返回的执行结果接着运行。

所以函数内部声明的指针，会随着函数调用完被栈自动释放掉。也就是说如果在函数内部通过malloc函数申请私有堆并声明一个指针后，如果该私有堆空间没有被及时的释放，当函数调用完后，该私有堆的指针便被栈释放后消失不见了，没有指向该堆的指针，又怎么能找到该堆呢。

在某些情况下，似乎不调用free（）释放内存是合理的。 例如，您的程序寿命很短，将很快退出； *在这种情况下，当进程终止时，操作系统将清理所有分配的页面，因此，不会发生内存泄漏。* 尽管这肯定是“有效的。但养成这种习惯可能是个坏习惯。

如果程序在退出之前忘记释放几兆字节，则操作系统将释放它们。 但是，如果您的程序一次运行数周，并且程序内部的循环忘记每次迭代都释放几个字节，则可能会发生巨大的内存泄漏，除非您定期重新启动它，否则它将吞噬计算机中的所有可用内存。即使程序最初用于某个大型任务，即使很小的内存泄漏也可能是很糟糕的。甚至于程序因为可用内存不足而崩溃。

程序运行时候：

进程中的每个地址可以处于三种状态。

1.占用状态(MEM_COMMIT)：线性地址被程序使用已经映射到实际物理内存中，已提交状态。

2.自由状态：没有映射到实际的物理内存，线性地址当前也没被程序使用。

3.保留状态(MEM_RESERVE)：保留一个线性地址，现在不用以后用，线性地址没有被映射到物理内存中。就像是占座。

LIB静态链接库：

我将代码编译成各种obj文件后，采用工具将其全部链接到一个函数库便是lib静态链接库，当程序需要调用函数方法的时候，链接器便会自动到lib链接库中寻找对应函数的实体并将其加载到程序文件中再进行编译。所以便导致了很多问题。

1.当lib内的函数实体需要更新的时候，就需要把之前编译过的程序再重新编译一次，一两个还好，如果几百上千一万来个呢？灵活性差。

2.每个程序将函数实体从lib库链接进来并存入内存中，那样的话内存中如果同时存在多个程序都链接了此函数实体，那内存中就会有很多空间放的都是该函数实体，编写同样的代码功能都要图省事将其封装成函数方法，那这种不同的空间却放的一模一样的函数实体怎么能忍呢？

DLL动态链接库：

基于lib静态链接的库的缺点发明出来的动态链接库，也就是DLL,与静态链接库一样，包含了各式各样的函数实体,但是DLL并不会在程序编译时将函数实体插入到程序中，而是当程序执行时windows会把相关联的动态链接库也载入到内存中，然后等着需要调用它的程序主动来找它。windows便会运用分页机制，将DLL函数实体映射到程序的虚拟空间中。

两者区别：

1.DLL与lib最直观的区别便是，LIB静态链接库是将函数实体直接安装到了程序身上，程序也会因为该实体的加入体积变大一些。

DLL则是将函数实体交到了程序手上，程序用完扔了就好了。并不会影响程序的体积，积少成多如果成千上万的程序都需要调用函数实体，DLL比LIB节省的空间还是挺多的吧。体积小了内存加载程序的速度也快，这性能不也就随之提升不少？

还有就是DLL因为是在程序执行时才会与程序进行链接。当我需要对函数实体进行更新时，只需要更改那一份DLL就行了。省不 少事。

Windows系统的权限划分：

在windows系统中有0,1,2,3四个权限等级，0为最高。3为最低。平常使用的来宾账户、普通账户、即使是管理员账户，都是在特权级3的基础上再细分的，而内核、驱动变是0特权级别的进程。

文件映射内存：

内存映射文件是由一个文件到一块内存的映射,使进程虚拟地址空间的某个区域与磁盘上某个文件的部分或全部内容的建立映射。. 建立映射后，通过该区域可以直接对被映射的磁盘文件进行访问.而不必执行文件I/O操作也无需对文件内容进行缓冲处理。. 就好像整个被映射的文件都加载到了内存一样，因此内存文件映射非常适合于用来管理大文件。. 内存映射文件对程序的提速，只在处理大文件或非常频繁的文件读写操作时效果才明显。. 通过内存映射，相当于将磁盘上的文件所在空间建立成一块虚拟内存，程序访问时可按内存的方式进行，省去了普通io方式的一些环节，其实真正要读写操作时，会进行换页，将这些个“虚拟内存”读到物理内存中。. 总之，内存映射文件是应用虚拟内存的技术来达到加速处理的.

内存映射文件读写文件的步骤：

  1. 调用CreateFile打开想要映射的文件，得到hFile。

  2. 调用CreateFileMapping函数生成一个建立在 CreateFile函数创建的文件对象基础上的内存映射对象，得到hFileMap。

  3. 调用MapViewOfFile函数把整个文件的一个区域或者整个文件映射到内存中。得到指向映射到内存的第一个字节的指针lpMemory 。

  4. 用该指针来读写文件。

  5. 调用UnmapViewOfFile来解除文件映射，传入参数为lpMemory 。

  6. 调用CloseHandle来关闭内存映射文件，传入参数为 hFileMap。

  7. 调用CloseHandle来关闭文件，传入参数为hFile。

  • 邮槽（Mailslot）也称为邮件槽，它是 Windows 提供的一种用来实现进程间通信的手段,是单向数据传输的服务。
  • 邮槽的弊端：邮槽是单向通信。服务器端只能读取，客户端只能写入。如果要双向通信就需要服务端写一个邮槽，客户端在写一个邮槽。创建两个邮槽。
  • 服务端和客户端的区分：创建邮槽的为服务端，已存在的邮槽的哪一端。
  • 客户端在使用邮槽发送数据的时候只有当数据的长度<425字节的时候，才能被广播给多个服务器，如果消息长度>425字节，邮槽不支持广播通信。
  • 邮槽最大的缺点是只允许从客户机到服务器，建立一种不可靠的单向数据通信。而邮槽最大的一个优点在于，它们使客户机应用能够非常容易地将广播消息发送给一个或多个服务器应用。
  • 将邮槽理解为文件系统，那么其实就是创建文件、写入文件和读取文件的关系。两台计算机使用邮槽通信，其实就是在服务端上创建一个文件供写入，当其他计算机（或进程）需要给它发送消息时，就是打开这个文件，将内容写入。当目的计算机进程发现文件内容非空，就读取文件进行分析显示。
实现流程：
  • 调用CreateMailslot函数，将创建邮槽的请求传递给内核的系统服务。
  • 请求传递给NtcreatMailslot函数，这个函数会到达底层的邮槽驱动程序。
  • 底层驱动程序msfs.sys，然后一些创建邮槽的工作就交给邮槽驱动程序来完成了.

  • 对于写入网络上其他主机的邮槽文件，就是 \\目标计算机\MailSlot\路径\文件名。

  • 定工作组/域发送，可以使用\工作组/域名\MailSlot\路径\文件名

左与右

C++对于左值和右值没有标准定义，但是有一个被广泛认同的说法：

  • 可以取地址的，有名字的，非临时的就是左值；
  • 不能取地址的，没有名字的，临时的就是右值；

可见立即数，函数返回的值等都是右值；而非匿名对象(包括变量)，函数返回的引用，const对象等都是左值。

从本质上理解，创建和销毁由编译器幕后控制，程序员只能确保在本行代码有效的，就是右值(包括立即数)；而用户创建的，通过作用域规则可知其生存期的，就是左值(包括函数返回的局部变量的引用以及const对象)。

右值引用，自动回收，快速备份值。

中断请求级别 IRQL 标识出CPU当前所执行代码的重要程度：

驱动常用IRQL

正在运行的线程可以随时被中断打断，进入到中断处理程序。优先级高的中断可以打断优先级低的中断处理程序，进入更高级别的中断处理函数。

0 - 2级为软件中断

3 - 31级为硬件中断

PASSIVE_LEVEL
IRQL最低级别,代表CPU在正常执行,没有中断发生，在这个级别上，线程执行用户模式，可以访问分页内存及非分页内存，可以使用所有内核API。DriverEntry、DriverUnload、DispatchRead…..等分发函数都处于这个级别

APC_LEVEL
在这个级别上,只有APC级别的中断被屏蔽，可以访问分页内存及非分页内存。部分API

DISPATCH_LEVEL
这个级别，DPC(延迟过程) 和更低的中断被屏蔽，不能访问分页内存，所有的被访问的内存不能分页。因为只能处理非分页内存，所以在这个级别，能够访问的Api大大减少。特定内核API

对于高于或者等于DISPATCH_LEVEL级别的程序不能使用分页内存，必须使用非分页内存

互斥锁：

互斥锁的同步方式是悲观的。当有多个线程同时访问同一块资源对象时，当一个线程访问到，windows操作系统变会将资源锁定，禁止其他线程访问，只有访问到的线程才能继续访问，其他线程都会被阻塞休眠。当线程访问完后，操作系统会唤醒休眠的线程让其访问。

这样就会产生一个问题，如果都是读操作，对资源对象并不会发生改变的操作，就不用每次调用的时候都锁定资源。互斥锁会导致效率很慢。

自旋锁：

资源对象会有一个状态值，为1表示没空，其他线程无法访问，每个线程也会有两个状态值，old value = 0 ；new value = 1；当资源对象被线程访问到时，线程会用0去比对资源对象状态值。比对成功就进去访问。并把资源对象值改成1，如果比对失败，就会原地自旋踱步不断的去比对状态值，直到比对成功进去访问。

使用内存：

操作系统有一个内存管理器，是给应用程序获得内存提供服务的，我们用的malloc()是从堆管理器申请内存的，堆管理器像个批发中间商一样，先从内存管理器申请一堆内存，然后当在堆中申请内存就不必跑那么远找内存管理器了，堆管理就给你了

后备列表（旁视列表）:

Windows提供了一种基于lookaside list的快速内存分配方案，不同于一般的使用ExAllocatePoolWithTag()系列函数的内存分配方式。每次从后备列表中分配固定大小的内存。系统构造了两个后备链表：ExNPagedLookasideListHead和ExPagedLookasideListHead，分别用于非分页内存和分页内存。类似于一个缓存，申请固定大小内存释放后，再申请，会发现地址还是那个地址。

后备列表列表头是一个 LIST_ENTRY 结构，因此：整个后备列表链是一个双向链表。ExNPagedLookasideListHead 和 ExPagedLookasideListHead 是内核的全局变量，由系统初始化。
