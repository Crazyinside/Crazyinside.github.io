逆向工程

逆向意义：

当我们非常想知道某个软件的某些功能如何实现的，而手头没有资料的时候，或者是对各种未开源的SDK或者软件进行安全审计，挖掘其中漏洞的时候。那么就需要对软件进行逆向工程，通过阅读其反汇编代码来推断其数据结构、体系结构和程序设计思路。

函数调用约定：

函数（子过程）是最早实现软件复用的机制，那么要调用现有的函数的一个最基本的问题就是如何去调用它，包括怎么传递参数，怎么接受计算结果，如何维持程序的上下文以及怎么平衡堆栈等等，因为要调用的函数可能是不同人使用不同语言和不同编译器开发的，所以为了解决以上问题，那么在发起调用和被调用的函数之间建立一种契约（协议）。这就是函数调用约定。

C调用约定__cdecl

C/C++程序的普通函数所使用的默认调用约定，完全使用栈来传递参数，调用者发起调用之前将参数从右到左压入栈中，并负责函数返回后调整栈指针，清除压入栈的参数，外部平栈，支持可变参数。

标准调用约定__stdcall

跟C调用类似，只不过参数清理由被调用者清理，内部平栈，这种方式不支持可变参数

快速调用约定 __fastcall

在X86下，对于前2个（从左到右）长度不超过32位的参数，使用ECX寄存器和EDX寄存器来传递（从右到左），其他参数使用栈传递（从右到左），内部平栈。

this调用约定 __thiscall

C++程序中的成员函数(类方法)默认使用this调用约定，this指针放入ECX寄存器传递，内部平栈。

X64调用约定

X64系统通常只用一种变形的FastCall，使用寄存器传递前4个参数，其他参数使用栈来传递，类似于cdecl和fastcall的混合版。RCX RDX R8 R9用来传递64位或短于64位的整数类型或指针类型，XMM0-XMM3用于浮点类型，结构类型会被自动当做指针类型进行传递，内部平栈。

局部变量和全局变量

局部变量为函数内部定义的变量，其存放的内存区域为栈区，生命周期为进入函数时分配，函数返回时释放

全局变量的地址在编译期就会固定下来，一般会用固定的地址去访问全局变量，先定义的在低地址，后定义的在高地址，根据此特征可以还原全局变量在源代码中的定义顺序

数组

相同数据类型的集合，以线性方式连续存储在内存中（从低地址到高地址）

一维数组的寻址公式：数组元素地址 = 数组首地址 + sizeof(数组类型) x 下标

二维数组的寻址公式：数组元素地址 = 数组首地址 + sizeof(一维数组类型) x 下标1 + sizeof(数组类型) x 下标2

[数组首地址 + 寄存器 x n]

case表优化

以下代码中，case语句的标号为一个数值为17的有序序列，按照if..else if 转换规则，则会将17的数值依次比较依次，从而得到分支选择结果，效率太低。

优化：

当case 大于等于6时候，且case值的间隔比较小时，编译器就会采用case表的方式实现switch语句。

首先代码生成时会为case语句制作一个case地址表数组，数组中保存每个case语句块的首地址，并且下标以0开头，在进入switch后先进行一次比较，检查输入的数值是否大于case值的最大值，对于没有case对应的数值，编译器以switch的结束地址或者default语句块的首地址填充对应的表格项。

然后把case的值当成数组下标进行跳转，避免使用if 从而提高性能

平衡判定树优化

当最大case值与最小case值之差大于255时，则会采用判定树优化，将每个case值作为一个节点，从节点中找出中间值作为根节点，以此形成一颗平衡二叉树，以每个节点为判定值，大于和小于关系分别对应左子树和右子树，从而提高查询效率。

花指令

花指令的英文是 junk code ，也就是垃圾代码。实际上花指令的确是一些不影响程序逻辑的 垃圾 机器码，它存在的唯一意义就是干扰反汇编引擎和人肉分析。

一般分析一个程序可以有动态和静态两条路，动态一般指的就是调试或者别的运行时跟踪程序行为的方式了，除了调试器外就是抓取事件、日志、API调用记录、看内存数据等，比如有 Frida，还有内存搜索如CE。静态则是用各种工具在不实际运行程序的前提下，从程序文件源码里提取有用的信息。

广为人知的对抗静态分析的手段有这些：

1.混淆，把程序逻辑转换成更晦涩但等价的形式。

2.加花，对抗反汇编引擎，利用反汇编工具的算法缺陷、漏洞来迫使分析者必须花费大量时间处理错误的反汇编结果，让诸如控制流视图之类的工具失效。

这些对抗手段主要的目的都是 消磨耐心 和 拖延时间 ，抬高人肉分析的成本。

举个最简单的例子：E8是汇编 call 指令对应的机器码，但如果我刚好存储了一个数据是 E8 , 我知道，CPU作为执行者也知道，但是反汇编软件不知道，他们还是机械思维看到这个 E8 ，哦这里是call指令，它就会把这个数据属性的 E8 当成一个call指令并显示在反汇编GUI中让别人阅读的而也是个 call 指令，那坐在屏幕前的人很容易也被其误导当成个call指令，然后它就发现这个程序的代码它看不懂了或是看起来不像是恶意程序，从而达到免杀的效果。

花指令有两种类型：

  • 不可执行的花指令
  • 可执行的花指令

反汇编算法

目前，主要的两类反汇编算法是线性扫描算法(Linear Sweep)和递归行进算法(Recursive Traversal)

线性扫描算法的技术含量不高,它将整个模块中的每一条指令都反汇编成汇编指令，将遇到的机器码都作为代码处理，没有对所反汇编的内容进行任何判断。无法正确地将代码和数据分开，数据也被当成代码来解码,从而导致反汇编出现错误。这种错误将影响对下一条指令的正确识别，会使整个反汇编都出现错误。（这便是刚刚的例子）。

而递归行进算法是基于代码流分析的，不会无脑反汇编，而是按照代码可能的执行顺序来反汇编程序，对每条可能的路径进行扫描。优先反汇编“可达”的代码，当解码出分支指令后,反汇编工具就将这个地址记录下来,并分别反汇编各个分支中的指令。这种算法比较灵活,可以避免将代码中的数据作为指令来解码。但是可以通过可执行的花指令来欺骗

完整性校验

当软件被开发出来时，为了增加软件的安全性，防止被破解，通常情况下都会对自身内存或磁盘文件进行完整性检查，以防止解密者修改程序，我可以将exe与dll文件同时做校验，来达到相互认证的目的，解密者想要破解则比较麻烦，当使用的互认证越多时，解密者处理的难度也就越大。

文件完整性检验的原理就是文件发布时用散列函数计算文件的散列值,并将此值放在某处,以后文件每次运行时,重新计算文件的散列值，并与原散列值进行比较，以判断文件是否被修改了。

CRC算法：该算法全称为CRC32循环冗余校验码，其主要作用适用于校验数据的完整性，CRC32多为32位，可以对字符串进行CRC32转换，得到一个4字节的CRC32值。
1.对磁盘文件完整性校验

实现磁盘文件检测，我可以使用CRC32算法或者RC4算法来计算程序的散列值，以CRC32为例，其默认会生成一串4字节CRC32散列，我只需要计算后将该值保存在文件或程序自身PE结构中的空缺位置即可。

使用CRC32算法计算出程序的CRC字节，并将其写入到PE文件的空缺位置（也可以把CRC32值写入一个单独的文件，运行时再读取比较），这样当程序再次运行时，来检测这个标志，是否与计算出来的标志一致，来决定是否运行程序，一旦程序被打补丁，其crc32值就会发生变化，一旦发生变化，程序就废了，绕过的方式则是，

1.动态打补丁，

2.找到crc32算法位置将跳转nop掉，

3.直接手动计算出crc32并覆盖到原始程序的crc算法位置。

2.内存映像完整性校验

磁盘文件完整性校验技术可以有效抵抗解密者直接修改硬盘文件，但对内存补丁没有效果，无法抵御对方动态修改机器码，为了防止解密者直接对内存打补丁，我需要在硬盘校验的基础上，增加对内存关键代码数据的校验

通常程序中至少包括了代码段，数据段，而数据段中所存储的数据是经常会发生变动的，例如我的全局变量，静态变量等都会默认存储在数据段，而代码段则不会发生变化，我在检验时只需要注重.text内存段中的数据完整性即可，针对内存的校验同样可以抵御调试器的CC断点，该断点原理就是在下端处写入int3指令，同样可以检测得到，但是硬件断点不会影响校验值，因为其实现是使用的DR3-DR0寄存器，没有改变内存数据。

在实际应用中，为了提高效率，有时我只需要保护其中一个片段代码就好.

通过使用磁盘校验结合内存校验两种方式综合保护，可以极大的提高软件的安全性，但是依然可以绕过：找到哪儿跟全局变量将其修正为正确的值即可，同样的也可以更暴力一些直接将判断条件改掉。其归根结底的原因是因为进程能轻易被用户进行反汇编调试，跟漏洞的本质便是用户参数可控一样。

但是自己的程序运行在别人的计算机上，别人反汇编不反汇编你控制不了，也把握不住，所以只能从心理上施加压力，例如花指令混淆，对代码加解密，或是反调试，杀人诛心，让其直接放弃，或是让其觉得没有。
